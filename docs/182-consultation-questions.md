## ❓ 질문 1 — 신규 가입자만 검증 (옵션 A)

**Q.** 신규 가입자만 만 14세 검증 절차를 적용하고, 기존 사용자는 점진적으로 재인증하는 정책이 PIPA §22-2 위반 **지속 상태** 인지, 아니면 **합리적 시정** 으로 인정되는지요?

**보충 정보 (질문관 요청 시)**:
- 베타 단계, 결제 미연동
- 신규 가입은 오늘부터 14세 검증 100% 적용
- 기존 약 N명 사용자는 birth_date NULL 상태 (재인증 절차 준비 중)

---

## ❓ 질문 2 — 단계적 제재 적정성

**Q.** 기존 사용자 대상 단계적 제재 일정이 적정한지요?
- D+0 ~ D+14: 자발적 birth_date 입력 안내 (이메일 + 로그인 모달)
- D+14 ~ D+30: 비입력 사용자 일부 기능 정지 (게시·미션 진행 제한)
- D+30 ~ D+60: 비활성 처리 (로그인 차단, 데이터 보존)
- D+60 ~ D+90: **개인정보 파기** (auth.users + public.users 삭제)

특히 **30일 / 60일 / 90일** 분할이 PIPA 가이드라인상 적정한지, 더 짧거나 길어야 하는지요?

## ❓ 질문 3 — 소셜 로그인 unverified 윈도우

**Q.** 소셜 로그인(카카오·구글·LINE) 직후 birth_date 입력 화면 (`/auth/complete-profile`) 까지 **수 초간 (auth.users 만 존재 / public.users 는 birth_date NULL)** 되는 unverified 윈도우가 PIPA 상 허용되는지요?

**보충**:
- OAuth callback → 추가 정보 입력 페이지로 즉시 redirect
- 14세 미만 입력 시 service-role API 로 auth.users 도 즉시 삭제 (cascade)
- 통상 < 10초 윈도우. 사용자가 입력 안 하고 떠나도 다음 로그인 시 재요구

대안: OAuth callback 단계에서 모든 신규 가입자를 일단 가입 거부 후 재로그인 요구. UX 큰 손해.

## ❓ 질문 4 — 의심 정황 SQL 점검 자체의 법적 의미

**Q.** 14세 미만 의심 정황을 SQL 로 read-only 조회 (닉네임 패턴, 평일 학교 시간대 활동) 하는 것 자체가 **사업자 의무 발생의 trigger** 인지요?

즉, "점검을 한 시점부터 모든 의심 사용자를 차단할 의무가 발생" 하는 건지, 아니면 "법률 자문 결과를 받기 전 단계" 로 SQL 실행이 가능한지요?

**보충 (제도 한계 명시)**:
- AI 이미지 분석 안 함 (PIPA 민감정보 회피)
- MEMORIES 사진 분석 절대 안 함
- 학교 시간 활동은 방학·재택수업·자유직업 가능성 있어 **단정 불가**
- 결과는 수치만 추출 (개별 사용자 ID 격리 X)

## ❓ 질문 5 — 법정대리인 동의 시스템 부재

**Q.** 베타 단계에서 법정대리인 동의 시스템 미구축 상태로, 14세 미만 발견 시 다음 옵션만 제공하는 것이 **시정 노력으로 인정** 되는지요?

- ① 14세 이상 도달 시까지 가입 차단 (`/auth/age-restricted` 페이지 안내)
- ② 부모 가입 메일 안내 (`hello@cloudwithyou.com`)
- ③ 기존 14세 미만 사용자 발견 시 자발적 탈퇴 또는 강제 파기

법정대리인 동의 시스템 (서면·전자서명) **미구축이 베타 단계 면책 사유** 가 되는지, 아니면 **지금 즉시 구축해야** 하는지요?
